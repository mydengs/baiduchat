## 一、产品概览

**产品名称**：百度文心（Baidu ERNIE / 文心一言）  
**访问地址**：https://chat.baidu.com  
**定位**：百度推出的大语言模型驱动的 AI 对话产品，支持多模型切换，集成联网搜索能力。

### 1.1 截图分析（v1.0）

截图展示的是文心首页「新建对话」状态，关键信息如下：

**界面元素**：
- 顶部欢迎语："下午好，今天从哪里开始？"
- 输入框占位提示："父亲节是哪一天"（用户尚未提交）
- 底部工具栏：语音输入、附件上传、图片上传、发送按钮
- **模型选择下拉框**（已展开）：

| 模型名称 | 描述 |
|----------|------|
| DeepSeek-V4 Pro ✓（已选） | 编程更强，推理更优 |
| DeepSeek-V4 Flash | 响应更快，适合日常问答 |
| DeepSeek-R1 | 深度思考，适合复杂问题 |
| 文心 5.1 | 文心最新模型，搜索推理能力强 |

- **思考模式**：开关已启用（蓝色）

**值得注意**：页面同时集成了第三方模型（DeepSeek 系列）与自研模型（文心 5.1），体现了百度平台化的产品策略。

## 二、URL 结构分析

### 2.1 完整格式

```
https://chat.baidu.com/search/{对话ID}?enter_type=home_tab
```

**示例**：
```
https://chat.baidu.com/search/7973619417296619530?enter_type=home_tab
```

### 2.2 各部分规则解析

| 部分 | 值（示例） | 含义 |
|------|-----------|------|
| `域名` | `chat.baidu.com` | 文心对话系统主域 |
| `路径前缀` | `/search/` | **搜索模式对话**专属路径，代表启用了「大模型回答 + 联网搜索」 |
| `对话ID` | `7973619417296619530` | 后端生成的唯一会话标识，数字串，绑定本次对话的全部上下文 |
| `查询参数` | `enter_type=home_tab` | 来源埋点，表示从首页标签页进入，仅用于统计，不影响对话功能 |

### 2.3 路径类型对比

| 路径 | 模式 | 说明 |
|------|------|------|
| `/search/` | 搜索对话 | 大模型 + 联网搜索（当前截图类型） |
| `/chat/` | 纯创作对话 | 纯大模型生成，不联网 |

## 三、新建对话的完整业务流程

点击「开启新对话」触发的完整链路：

```
步骤 1：用户前端触发
  └─ 用户在首页点击「开启新对话」
  └─ 前端向后端发起新对话创建请求

步骤 2：后端生成会话
  └─ 后端生成唯一会话 ID（如 7973619417296619530）
  └─ 初始化对话上下文容器（准备存储多轮问答）

步骤 3：前端路由跳转
  └─ 前端将会话 ID 拼入路由，生成最终 URL
  └─ 页面跳转到新对话窗口，完成初始化

步骤 4：后续交互就绪
  └─ 用户输入问题后，才会调用 conversation 和 impush 接口
```

## 四、核心接口关系与协作流程

三类组件分工明确，共同完成完整的 AI 对话体验：

### 4.1 组件职责总览

| 组件 | 职责定位 | 触发时机 |
|------|---------|---------|
| **新建对话 URL** | "对话容器"，完成初始化和身份标识 | 用户点击新建对话时 |
| **`conversation` 接口** | "交互核心"，处理用户提问、流式返回大模型回答 | 用户在对话窗口提交问题后 |
| **`impush` 接口** | "实时通道"，长连接，推送会话状态/系统通知 | 对话创建完成后持续维持 |

### 4.2 完整协作时序

```
用户端
  │
  │── 点击新建对话 ──────────────────────→ 页面路由(新对话URL)
  │                                              │
  │                                              ├── 初始化本次对话上下文
  │                                              └── 返回带唯一ID的URL
  │
  │── 输入问题 ───────────────────────────→ conversation 接口
  │                                              │
  │ ←── 流式返回大模型生成的回答 ─────────────────┘
  │
  │ ←── [持续长连接] 会话状态 / 系统消息 ──── impush 接口
  │         (对话超时、多端同步、系统提示等)
```

### 4.3 关联机制说明

- **会话 ID 贯穿全程**：`conversation` 和 `impush` 的每次请求，都需携带 URL 中的会话 ID，让后端识别对话归属并加载对应上下文。
- **impush 优先建立**：对话 URL 初始化完成后，前端即与 `impush` 建立长连接，保障实时通知能力先于用户提问就位。
- **流式输出特征**：`conversation` 接口采用流式（Streaming）返回，支持逐字输出的打字机效果。

## 五、conversation 请求载荷参数分析（v1.1）

### 5.1 从 DevTools 截图捕获的 Payload 字段

两张截图中 DevTools「负载」面板展示了 `conversation` 接口的 **请求体（Request Body）** 部分字段：

```json
{
  "message": {
    "inputMethod": "chat_search",
    "isRebuild": false
    // ...（更多字段待捕获）
  },
  "setype": "csaitab",
  "rank": 14
}
```

截图二中出现了 `rank: 15`，与截图一的 `rank: 14` 形成对比，说明该字段在同一会话中递增。

### 5.2 已知参数解析

| 参数名 | 捕获值 | 类型 | 含义分析 |
|--------|--------|------|---------|
| `message.inputMethod` | `"chat_search"` | string | 对话模式标识，`chat_search` = 搜索模式对话（区别于纯创作模式） |
| `message.isRebuild` | `false` | boolean | 是否为重新生成/再次回答的请求；`false` = 正常首次提问；`true` = 用户点击"重新生成"时 |
| `setype` | `"csaitab"` | string | **场景类型标识**（推测）：`csa` = Conversational Search AI，`itab` = 界面标签页入口；用于后端区分不同场景下的调用路径 |
| `rank` | `14` / `15` | integer | **消息序号**（推测）：在同一会话内单调递增，用于标识本条消息在多轮对话中的位置；可能用于后端顺序管理与去重 |

### 5.3 关键推断：`rank` 字段的业务意义

截图一 `rank: 14` → 截图二 `rank: 15`，说明：
- `rank` 不是随机值，而是**会话内消息的全局计数器**
- 每次用户提交一条新消息，`rank` +1
- 后端可用此字段判断消息顺序，防止乱序或重复处理
- 可能同时用于前端重试逻辑：若重试请求 `rank` 相同，后端可识别为幂等请求

### 5.4 待补全的 `message` 对象字段

当前 `message` 对象仅捕获到两个字段，完整结构预计还包含：

| 待确认字段 | 推测含义 |
|-----------|---------|
| `content` / `query` | 用户实际输入的问题文本 |
| `sessionId` / `chatId` | 会话 ID（即 URL 中的数字串） |
| `modelType` / `model` | 当前选中的模型标识（如 DeepSeek-V4 Pro） |
| `thinkMode` | 思考模式开关状态 |
| `contextMessages` | 历史上下文消息列表 |
| `searchEnabled` | 是否开启联网搜索 |

## 六、impush 接口初步分析（v1.1）

## 七、第三方客户端对接规划（v1.1）

### 7.1 目标

能从第三方客户端（如 Postman、Python 脚本、自建前端）直接调用 `conversation` 接口，发送问题并获取流式回答。

### 7.2 需要收集的完整信息

#### A. 认证信息（最关键）

百度系产品使用 Cookie 做身份鉴权，登录态核心字段：

| Cookie 名 | 作用 | 备注 |
|-----------|------|------|
| `BDUSS` | **主鉴权票据**，最重要 | 登录后由百度账号系统签发 |
| `STOKEN` | 安全 Token，部分接口校验 | 与 BDUSS 配套使用 |
| `BAIDUID` | 设备标识 | 首次访问自动生成 |
| `BDORZ` | 登录态辅助标识 | 配合 BDUSS 使用 |

> **获取方式**：浏览器登录文心后，DevTools → Application → Cookies → `chat.baidu.com`

#### B. 请求头（Headers）

| 字段 | 值（示例） | 说明 |
|------|-----------|------|
| `Content-Type` | `application/json` | 请求体格式 |
| `Referer` | `https://chat.baidu.com/search/{conv_id}` | 防盗链校验，必须携带（conv_id 即 URL 中的数字串） |
| `Origin` | `https://chat.baidu.com` | 跨域来源校验 |
| `Cookie` | `BDUSS=...;STOKEN=...;BAIDUID=...` | 身份鉴权 |
| `User-Agent` | 浏览器 UA 字符串 | 伪装正常浏览器请求 |
| `X-Requested-With` | 待确认 | 可能存在自定义鉴权头 |

## 十、conversation 接口详解：统一入口与纯文字/图文差异（v1.3）

### 10.1 统一入口设计

`conversation` 是文心一言网页端的**统一对话入口**，所有请求类型都走这一个端点：

```
纯文字提问  ──┐
图文混合提问 ──┼──→  POST /aichat/api/conversation  ──→  后端按内容分发到对应模型
语音转文字  ──┘
```

前端**不区分**提问类型选择不同接口，统一封装后由**后端根据请求内容自动路由**。

### 10.2 纯文字请求的处理流程

```
① 前端封装：用户文字 → message 字段 + conv_id 会话标识 → 构造 POST 请求体
② 后端路由：识别为纯文字 → 路由到纯文本大模型（如 ERNIE 4.0 / 文心 5.1）
③ 流式返回：Event-Stream 分块输出 → 前端逐字渲染（打字机效果）
```

纯文字请求**无需手动指定模型参数**，后端自动匹配。

### 10.3 纯文字 vs 图文混合：请求体差异对比

| 对比项 | 纯文字请求 | 图文混合请求 |
|--------|----------|------------|
| `message.content` | 仅包含文本字符串 | 包含文本内容 |
| 图片字段 | **无** | 额外携带 `image_base64` 或 `image_url` |
| 后端路由模型 | 纯文本大模型（ERNIE 4.0 / 文心 5.1 等） | 多模态 VL 大模型（ERNIE-VL） |
| 请求体其余字段 | 相同（`conv_id`、`setype`、`rank` 等） | 相同（`conv_id`、`setype`、`rank` 等） |

## 十三、Brotli 压缩处理（第三方客户端实现要点）v1.4

响应头 `Content-Encoding: br` 说明服务端返回的 SSE 数据流是 **Brotli 压缩**的。这是第三方客户端实现的重要技术细节。

### 13.1 Python 客户端处理方案

```python
import httpx    # 推荐，原生支持 Brotli 解压
# 或
import requests # 需要额外安装 brotli 库: pip install brotli

# 请求时需要告知服务器客户端支持的压缩格式
headers = {
    "Accept-Encoding": "gzip, deflate, br",  # 声明支持 Brotli
    # ... 其他必要 headers
}
```

> `httpx` 是首选，内置 Brotli 支持，且对 SSE 流式读取更友好。

### 13.2 Brotli 支持安装

```bash
pip install httpx[brotli]   # httpx + brotli 解码器
# 或
pip install requests brotli  # requests + brotli 解码器
```

## 二十一、impush 接口完整分析（v1.6 实测确认）

> ✅ 来源：DevTools 负载面板 + 响应面板实测截图，可信度最高。

### 21.1 协议类型确认：长轮询（Long Polling）

之前推测 impush 可能是 WebSocket 或 SSE，**实测确认为 HTTP 长轮询**。

| 特征 | 观察结果 |
|------|---------|
| 协议 | 普通 HTTP POST（非 WebSocket 握手，非 SSE 流） |
| 交互模式 | 客户端主动发请求 → 服务端持有或立即响应 |
| 响应格式 | 标准 JSON（非分块流式） |
| 连接方式 | 请求-响应完成后断开，客户端再次发起 |

### 21.2 impush 请求体（实测）

```json
{
  "call_from":  "aitab",
  "logid":      "60629982750896152702",
  "pa_uid":     17592352028333,
  "msg_readed": 1
}
```

| 字段 | 类型 | 含义 |
|------|------|------|
| `call_from` | string | 调用来源标识，`"aitab"` = AI 对话标签页 |
| `logid` | string | **消息游标**：客户端上一次收到的 logid，告知服务端"我已读到这里" |
| `pa_uid` | number | **用户账号 ID**（Permanent Account UID），百度账号唯一数字 ID |
| `msg_readed` | number | 已读标志，`1` = 已处理上一条消息 |

### 21.3 impush 响应体（实测）

```json
{
  "status":  0,
  "message": "",
  "logid":   "80382986397376372211"
}
```

| 字段 | 含义 |
|------|------|
| `status` | `0` = 正常，无新消息；其他值 = 有推送或错误 |
| `message` | 推送内容，当前为空（无通知） |
| `logid` | **新的游标**，客户端下次请求时携带此值 |

### 21.4 impush 工作机制

```
客户端                              impush 服务端
  │                                      │
  │─── POST {logid: "A", msg_readed:1} ──→│
  │    (服务端持有连接，等待推送事件)        │
  │                                      │  等待...
  │←── {status:0, message:"", logid:"B"}─│  (无事件时立即/超时返回)
  │                                      │
  │─── POST {logid: "B", msg_readed:1} ──→│  再次轮询
  │←── {status:X, message:"...", logid:"C"}│  (有事件时携带数据)
```

**关键设计**：`logid` 充当游标，每次响应返回新的 `logid`，客户端用新值发起下一轮轮询，实现持续的消息跟踪。

### 21.5 pa_uid 的含义与获取

`pa_uid: 17592352028333` 是百度账号的**永久数字用户 ID**。

获取方式（几个途径）：
1. 从 `impush` 接口本身的请求中观察到（已在截图中获得）
2. 从百度账号 API 接口获取（需要登录状态）
3. 可能在 Cookie 或 localStorage 中也有存储

> ⚠️ 构造 impush 请求时，`pa_uid` 是必填字段，需要提前获取当前账号的数字 ID。

| 事件 | 作用 |
|------|------|
| `event: basedata` | 返回基础信息：登录态、query、baiduid、lid、日志信息 |
| `event: ping` | 心跳 |
| `event: message` | 返回状态、搜索信息、增量回答、结束信号、推荐追问等 |

### 23.5 增量回答解析路径

真正的模型回答文本出现在：

```text
data.message.content.generator.data.value
```

每个 `event: message` 中的 `data:` 是 JSON。解析逻辑：

```python
chunk = obj["data"]["message"]["content"]["generator"]["data"]["value"]
answer += chunk
```

不是所有 `message` 都包含 `generator.data.value`，客户端必须容错：

- `waiting-resp` 阶段可能只有 `cacheStatus` 或 `searchQuery`
- `generating-resp` 阶段通常出现 `generator.data.value`
- `generate-complete` 阶段可能出现反馈区、推荐追问、最终统计

### 23.6 结束信号

主回答完成时出现：

```json
{
  "metaData": {
    "state": "generate-complete",
    "endTurn": false
  },
  "content": {
    "generator": {
      "isFinished": true
    }
  }
}
```

整轮对话最终结束时出现：

```json
{
  "metaData": {
    "state": "generate-complete",
    "endTurn": true
  }
}
```

因此客户端建议：

| 目标 | 判断方式 |
|------|---------|
| 停止拼接主回答 | `generator.isFinished == true` |
| 关闭本轮等待状态 | `metaData.endTurn == true` |
| 展示推荐追问 | `component == "questionClosely"` |

### 23.7 最小 SSE 解析伪代码

```python
answer_parts = []

for line in response.iter_lines():
    if not line or not line.startswith("data:"):
        continue

    obj = json.loads(line[5:])
    message = obj.get("data", {}).get("message", {})
    meta = message.get("metaData", {})
    generator = message.get("content", {}).get("generator", {})

    value = generator.get("data", {}).get("value")
    if value:
        answer_parts.append(value)

    if generator.get("isFinished") is True:
        final_answer = "".join(answer_parts)

    if meta.get("endTurn") is True:
        break
```

### 24.3 `X-Chat-Message` 真实结构

v1.8 cURL 证明：`X-Chat-Message` 不是完全不可读的二进制 token，而是一个**逗号分隔的摘要型动态头**，其中部分值为 URL 编码。

```text

  anti_ext:%7B%22inputT%22%3Anull%2C%22ck1%22%3A3922%2C%22ck9%22%3A1001%2C%22ck10%22%3A796%7D,
  enter_type:chat_url,
  re_rank:7,
  modelName:smartMode,
  sa:bkb
```

解码后等价于：

```json
{
  "query": "用户输入的问题",
  "anti_ext": {
    "inputT": null,
    "ck1": 3922,
    "ck9": 1001,
    "ck10": 796
  },
  "enter_type": "chat_url",
  "re_rank": "7",
  "modelName": "smartMode",
  "sa": "bkb"
```

关键结论：

| 字段 | 来源/对应关系 |
|------|--------------|
| `query` | 对应请求体 `message.query[0].data.text.query` |
| `anti_ext` | 对应请求体 `message.anti_ext` |
| `enter_type` | 对应请求体 `message.searchInfo.enter_type` |
| `re_rank` | 对应请求体 `message.searchInfo.re_rank`，通常与外层 `rank` 一致 |
| `modelName` | 对应请求体 `message.searchInfo.usedModel.modelName` |
| `sa` | 对应请求体 `message.searchInfo.sa` |

| `sub_session.ori_lid` 从第二轮起绑定某个子会话起点/首个回答 lid | 更符合第三轮样本 |
| `sub_session.re_rank` 表示子会话内轮次 | 仍成立 |

目前更合理的状态模型：

```text
root_lid / searchInfo.ori_lid = 页面根会话 ID，保持不变
sub_session.ori_lid = 子会话入口/首个回答 lid，从第二轮开始保持稳定
sub_session.re_rank = 子会话轮次，2、3、4... 递增
rank / searchInfo.re_rank = 全局正式提问轮次，7、8、9... 递增
```

### 26.3 `aitabserver.tk` 与 `conversation.chat_token` 关系修正

第三轮中，最靠近正式 `conversation` 的 `aitabserver` 请求 URL 带：

```text
tk=ZGRmZWIxYWJ8ZjA2MGE2MTA1MTRmNmNmNmY1NmUzNThjMmYzZjVkMjR8MTc4MjM5MjQ3NTE2MHw3Njk4MzczNzk2MzkzNTc1MzEz-7698373796393575313-3
```

随后 `conversation` 请求体中带：

```text
chat_token=ZGRmZWIxYWJ8ZjA2MGE2MTA1MTRmNmNmNmY1NmUzNThjMmYzZjVkMjR8MTc4MjM5MjQ4MDM2OHw3Njk4MzczNzk2MzkzNTc1MzEz-7698373796393575313-3
```

二者对比：

| 片段 | 是否相同 |
|------|----------|
| 前缀 `ZGRmZWIxYWJ8` | 相同 |
| 中间 hash `ZjA2MGE2...ZjVkMjR` | 相同 |
| 时间戳段 | 不同：`1782392475160` vs `1782392480368` |
| 页面/设备 lid 段 `7698373796393575313` | 相同 |
| 尾部 `-3` | 相同 |

结论：

| 判断 | 结论 |
|------|------|
| `aitabserver.tk` 是否直接等于 `conversation.chat_token` | **否，本轮样本不相等** |
| 二者是否同源/同族 | **是，高度同源** |
| `aitabserver` 是否“响应返回 chat_token” | **当前 cURL 只能证明请求 URL 携带 tk，未证明响应体返回 tk** |
| 是否可直接拿最近一次 `aitabserver.tk` 当 `chat_token` | **不建议，需实测验证** |

因此，关于 `chat_token` 来源的当前更准确表述是：

> `chat_token` 与提问前后的 `aitabserver.tk` 使用同一套 token 生成机制，包含相同的业务前缀、hash、页面级 lid 和尾部场景标记；但正式 `conversation.chat_token` 会使用更靠近发送时刻的新时间戳，因此不能简单认定 `aitabserver.tk` 等于 `chat_token`。

### 26.4 对用户提供推断的校正

用户补充的说法中，以下部分与抓包吻合：

- `chat_token` 不是从 Cookie 直接提取；
- `chat_token` 是短时动态凭证；
- 它与当前浏览器 Cookie / 页面会话上下文有关；
- 它每轮发送前后会刷新；
- `aitabserver.tk` 与 `conversation.chat_token` 存在强关联。

但以下部分仍需标记为待验证或修正：

| 说法 | 当前判断 |
|------|----------|
| `aitabserver` 响应中直接返回 `tk` | 未验证；目前只看到请求 URL 中带 `tk` |
| `aitabserver.tk` 就是后续 `conversation.chat_token` | 不准确；第三轮显示二者同源但不同值 |
| 未登录拿到 token 也必然不能正常 SSE | 待真正无 `BDUSS` 抓包验证 |

### 26.5 下一步抓包重点

要继续确认 `chat_token` 获取方式，需要：

1. 复制最靠近 `conversation` 前一条 `aitabserver` 的**响应体**，而不是只复制 cURL；
2. 在 DevTools 全局搜索 `chat_token` 或 token 前半段，看它是否存在于 JS 运行时数据、localStorage、sessionStorage、接口响应中；
3. 抓真正无 `BDUSS` / `BDUSS_BFESS` 的匿名窗口样本，确认匿名态是否返回模型 SSE。

### 28.3 `chat_token` 前端生成算法确认

JS 中存在明确函数：

```js
function getToken(query, context) {
  let q = getStrQuery(query || "");
  let lid = context.searchframeLid || Date.now();
  let token = context.token || "";
  let queryMd5 = SparkMD5.hash(q);
  return `${window.btoa(`${token}|${queryMd5}|${Date.now()}|${lid}`)}-${lid}-3`;
}
```

另一个类方法版本：

```js
getToken(query) {
  if (typeof query !== "string") {
    query = getStrQuery(query);
  }
  let lid = this.context.searchframeLid;
  let token = this.context.token;
  let queryMd5 = SparkMD5.hash(query);
  return `${window.btoa(`${token}|${queryMd5}|${Date.now()}|${lid}`)}-${lid}-3`;
}
```

因此，`chat_token` 的结构已经可以解释：

```text
base64(token + "|" + md5(query) + "|" + timestamp + "|" + searchframeLid)
-
searchframeLid
-
3
```

字段含义：

| 片段 | 来源 | 说明 |
|------|------|------|
| `token` | `context.token` / 页面基础数据 | 业务 token，可能来自页面初始化数据 |
| `md5(query)` | 前端 `SparkMD5.hash(query)` | 用户问题文本的 MD5 |
| `timestamp` | `Date.now()` | 生成 token 的毫秒时间 |
| `searchframeLid` | `context.searchframeLid` | 页面级/搜索框会话 lid |
| 尾部 `-3` | 固定场景标记 | 当前样本固定为 `3` |

这解释了此前 v1.10 的观察：

- `aitabserver.tk` 和 `conversation.chat_token` 同源；
- 它们可能拥有相同的 `token`、`md5(query)`、`searchframeLid`；
- 但只要生成时间不同，base64 内部的 `timestamp` 就不同，所以二者不会完全相等。

### 33.3 对客户端实现的直接意义

第三方客户端启动时不应优先依赖抓取初始化 fetch URL 来反推 token，而应：

1. `GET https://chat.baidu.com/`，保持同一个 Cookie Jar；
2. 从 HTML 中提取 `script[type="application/json"][name="aiTabFrameBaseData"]`；
3. JSON 解析出 `token`、`lid`、`chatParams.setype`、`userInfo.isUserLogin`；
4. 用 `token/lid` 生成后续 `history/list`、`guidewords`、`conversation` 所需 token；
5. 发送 `conversation` 时仍保持同一批匿名 Cookie。

这一步补上后，匿名态对接的启动链路已基本可程序化实现。

### 33.4 仍需注意

`aiTabFrameBaseData` 中还包含 `sample`、`experimentParams` 等实验开关字段。它们可能影响页面功能开关、引导词、模型胶囊展示等，但目前尚未证明是 `conversation` MVP 必填字段。第一版客户端可先只使用 `token/lid/chatParams/userInfo`，其余字段保留为调试信息。

- 同时出现的 `ztbox?action=zpblog...` 是下载/点击行为埋点，不是图片生成或下载鉴权接口。

### 37.1 长文结果组件

### 37.2 文档增量字段

`editor-workspace-viewer` 的核心路径为：

```text

```

典型结构如下：

```json

  "data": {
    "value": {
      "type": "text",
      "id": "7905173392360479737_0",
      "title": "《我在修仙界搞考核》小说大纲",
      "fileName": "穿越修仙小说大纲",

          "fileId": "7905173392360479737_0",
```

因此，客户端若要还原完整长文，需要：

4. 使用 `files[].isFinished == true` 或后续 `generate-complete/endTurn` 判断工作区文件生成结束。

### 37.8.1 Download 响应形态（v1.29）

最新抓包确认，`/aichat/api/download` 不同 `fileType` 的响应形态不同。

#### Markdown

Markdown 导出响应直接返回正文内容：

```json
{
  "status": 0,
  "message": "",
  "data": {
    "url": "",
    "isDeleted": false,
    "content": "Markdown 正文内容"
  },
  "logid": "8205383415858762948"
}
```

结论：

- `fileType=markdown` 时，`data.url` 为空；
- 正文在 `data.content`；
- 客户端需要自己将 `data.content` 保存为 `.md` 文件。

#### Word / PDF

Word 与 PDF 导出响应返回对象存储下载地址：

```json
{
  "status": 0,
  "message": "",
  "data": {
    "url": "https://.../conversation/aitab/docx/export/xxx.docx?...",
    "isDeleted": false
  },
  "logid": "..."
}
```

PDF 类似：

```json
{
  "status": 0,
  "message": "",
  "data": {
    "url": "https://.../conversation/aitab/docs/export/xxx.pdf?...",
    "isDeleted": false
  },
  "logid": "..."
}
```

结论：

- `fileType=docx` 时，读取 `data.url` 下载 `.docx`；
- `fileType=pdf` 时，读取 `data.url` 下载 `.pdf`；
- URL 指向百度对象存储/云存储地址，通常带有响应头控制参数，例如 `responseContentDisposition=attachment`；
- 浏览器自动下载就是拿到 `data.url` 后继续请求该文件地址。

### 37.8.2 docx/pdf 签名下载 URL 细节（v1.30）

Word 文档样本：

```text
https://wkbjcloudbos.bdimg.com/v1/convformat/aitab/md2word/{uuid}.docx
```

PDF 样本：

```text
https://wkbjcloudbos.bdimg.com/v1/convformat/aitab/docx/export/{hash}.pdf
```

两类 URL 都带有签名和响应控制参数：

| 参数 | 作用 |
|------|------|
| `responseContentDisposition` | 控制浏览器按附件下载，并指定文件名 |
| `responseContentType` | 指定文件 MIME 类型，如 docx 或 pdf |
| `responseCacheControl` | 控制缓存时间 |
| `responseExpires` | 响应过期时间 |
| `authorization` | BCE/BOS 签名授权信息 |
| `token` | 额外 JWT 风格下载 token |

已观测 MIME 类型：

```text
docx: application/vnd.openxmlformats-officedocument.wordprocessingml.document
pdf:  application/pdf
```

已观测路径差异：

```text
docx: /v1/convformat/aitab/md2word/*.docx
pdf:  /v1/convformat/aitab/docx/export/*.pdf
```

有效期线索：

- URL 中 `authorization` 包含签名时间和秒级有效期，例如 `/3000/host/` 或 `/3600/host/`；
- URL 中 `token` 末尾也包含过期时间戳；
- `responseExpires` 是下载响应层面的过期/缓存控制时间，不一定等同于签名 URL 可访问期限；
- 客户端应在拿到 `data.url` 后立即下载，不应长期缓存该 URL。

因此，代码实现应当：

```text
POST /aichat/api/download
  → 读取 response.data.url
  → 立即 GET 该 URL
  → 按 fileType 保存为 .docx 或 .pdf
```

#### 客户端保存规则

```text
if fileType == "markdown":
    content = response.data.content
    保存为 .md

if fileType in ["docx", "pdf"]:
    url = response.data.url
    GET url
    保存二进制文件
```

完整链路：

```text
点击下载菜单
  → GET /workspace/text 加载/确保阅读器页面
  → GET /aichat/api/canvas/search 定位工作区文件并返回正文/元信息
  → POST /aichat/api/download 按 fileType 导出
  → markdown 返回正文；docx/pdf 返回文件 URL
  → docx/pdf 再请求 URL 下载文件
```
